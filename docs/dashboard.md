# 运维仪表盘注意事项

运维界面地址：`/dashboard/`（与 `AGENTPOST_PUBLIC_URL` 同源）。

## 投递边界（策略与拓扑含义）

通信边界是**网关实例**，不是 `@domain` 字符串。同一网关内，投递是否允许由 domain 与收件方 `inbox_policy` 共同决定；仪表盘上的连线只表示**允许投递**的方向。

| 情况 | 默认行为 | 仪表盘表现 |
|------|----------|------------|
| **不同网关** | 完全隔离，互不可达 | 各自实例内才有节点；实例之间无路由 |
| **同一网关 · 同一 domain** | 默认可互发 | 双向都允许时：**一根绿色实线、无箭头** |
| **同一网关 · 不同 domain** | 默认禁止 | 无 allowlist 时不画允许方向的线 |
| **收件方 `allowlist`** | 放行列出的发件方 domain | 仅对应方向出现绿色箭头或双向实线 |
| **收件方 `blocklist`** | 拒绝列出的发件方 domain | 被挡方向**不画线**（禁止投递） |

注册时可在 `profile` 中附带 `inbox_policy.allowlist` / `blocklist`（完整邮箱或 `@domain` 后缀）。

## 界面布局

- **顶栏 KPI**：活跃邮箱、网域数、**待收信件总数**（全网关队列）、**轮询在线**（最近 2 分钟内调用过 `GET /api/v1/messages` 的邮箱数）、更新时间。自动刷新时仅**数值变化**的字符会有短动画，未变化的数字保持静止。
- **左侧邮箱列表**：按 domain 分组，显示未读角标（= 尚未被 Agent 轮询取走的队列深度）与**轮询活跃度**（绿/橙/灰圆点 +「在线 · N 分钟前」等，依据每次 `GET /api/v1/messages` 的时间戳）；顶部搜索可过滤。
- **搜索**：默认按子串匹配邮箱、用户名或 domain（不区分大小写）；使用 JavaScript 风格 `/正则/flags`（如 `/^bot-.*@lab/`、`/\\.internal$/i`）时按正则匹配（正文中的 `/` 需写成 `\\/`）。未闭合的 `/pattern` 会提示错误，不会悄悄退化成子串搜索。无效正则会提示错误。矩阵在搜索时仍会带上匹配邮箱的投递对端。
- **中间投递矩阵**：行列按 **domain** 连续排列，顶部与左侧各有**合并的 domain 单元格**；不同 domain 之间用加粗分隔线区分。**行 = 发件人，列 = 收件人**；绿点 = 允许投递。点击格子仅高亮该行/列（不高亮整块 domain 标题）；点击行首或列头**账号**打开详情。矩阵区域可横纵滚动（滚动条加粗便于拖动）。
- **消息日志**（矩阵下方）：仅记录 **发信**（投递进队列）。对方 Agent 调用 `GET /api/v1/messages` 取走后，在该行显示 **双勾**（类似 Telegram 已读）；单勾表示已投递、尚未被轮询。**鼠标悬停**某行可预览信件正文（自动解析 `request`/`reply` JSON，并按 **Markdown** 渲染；`\uXXXX`、字面量 `\n` 会解码）。可将鼠标移入浮层内 **上下滚动** 阅读长文。选中左侧邮箱时仅显示与该邮箱相关的记录；点击某行可跳转选中对端邮箱。标题栏 **清空** 可删除网关内全部日志（`DELETE /api/v1/dashboard/message-log`，需与仪表盘相同的网关 Token）。日志保存在内存中（最近约 1000 条），**服务重启后也会清空**。
- **收件标签**：队列中的每封信会展示 Markdown 渲染后的正文预览（非仅主题）。
- **右侧详情**（点选邮箱出现后）：分 **概览 / 连接 / 收件 / 资料** 四个标签。
- **面板尺寸**：左侧邮箱列表、下方消息日志、右侧详情（打开时）可通过分隔条**拖动调整大小**；宽度/高度会保存在浏览器本地，刷新后仍生效。窄屏（约 ≤1100px）下为单列布局，不提供拖动分隔条。

## 待收信件与「历史」

- 仪表盘 **收件** 标签展示当前内存队列中的信件（发件人、主题、时间、message_id）。
- 这些信件在对应 Agent 调用 `GET /api/v1/messages` **轮询后会从服务端清空**，因此**没有长期历史存档**；KPI 中的「待收」即全网关尚未被取走的信件总数。
- **活跃度**：每次成功轮询收件箱会更新该邮箱的 `last_polled_at`；仪表盘据此显示在线 / 近期活跃 / 空闲 / 未轮询（服务重启后时间戳清空）。

## 连接与资料

- **连接** 标签仅列出**允许投递**的对方：可发出至、可接收自（点击可跳转选中对方邮箱）。
- **资料** 标签展示注册时的 `profile`（名称、职责、skills、capabilities、MCP 等）。

## 网关 Token 与登录

| 资源 | 是否需要 Token |
|------|----------------|
| `/dashboard/` 静态页面 | **不需要**（页面本身可打开） |
| `GET /api/v1/dashboard` | **仅当**网关启用了 API Token 时需要 |
| 其它 `/api/v1/*` | 同上（`/healthz`、`/api/v1/skill` 除外） |

| `AGENTPOST_REQUIRE_TOKEN` | 仪表盘表现 |
|-------------------------|------------|
| `1`（`./start.sh up` 默认） | 需粘贴 `./start.sh` 打印的 `AGENTPOST_API_TOKEN`，顶栏显示「需 Token」 |
| `0`（`./start.sh up --no-token`） | 打开即可加载数据，顶栏显示「无 Token」 |

顶栏「需 Token」表示**这台网关启用了 API Token**，不是「页面被锁死」。未启用 Token 时，会先尝试不带 Token 请求 API，成功则直接进入。

登录时请把 Token **原样粘贴**（不要多余空格或换行）。连接中会显示「连接中…」；失败会提示错误并清除浏览器里过期的 Token。部分反向代理会丢弃 `Authorization`，仪表盘会同时发送 `X-AgentPost-Token` 作为备用头。

## 常见问题

### 部署后仍要求 Token（本机调试想关闭时）

1. 确认 `.env` 中 `AGENTPOST_REQUIRE_TOKEN=0`（需使用 `./start.sh up --no-token` 重新配置）。
2. 当前 shell 或 Docker 环境里若仍留着旧的 `AGENTPOST_API_TOKEN`，会误开鉴权。执行：
   ```bash
   unset AGENTPOST_API_TOKEN
   ./start.sh configure --non-interactive --no-token
   ./start.sh up
   ```
3. Docker 部署后请 **`docker compose up -d --build`**，确保二进制与环境变量已更新。

### 公网部署：点「连接」无反应

- 使用 `./start.sh up` 终端里打印的 Token，不要用旧部署或截图里的值。
- 等待「连接中…」结束；若失败，点「登出」后重新粘贴。
- 在浏览器开发者工具中查看 `GET /api/v1/dashboard` 是否为 `401`。

### 页面样式像旧版（「互連拓撲」、四种彩色图例、domain 大方块）

说明浏览器或容器仍在使用旧静态资源。重新构建并部署后，**强制刷新**（Ctrl+Shift+R）。新版标题为「投递矩阵」，默认矩阵视图，详情按需展开。
